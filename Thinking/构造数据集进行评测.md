# 如何进行Agent评测？
在评测体系上，我一般按链路拆成四层：检索层、决策层、生成层和系统层。检索层主要看 **Recall@K召回率、Top-1 evidence命中率**；决策层看**工具调用是否正确、参数是否正确**；生成层看**答案准确性、证据一致性、完整性和幻觉率**；系统层看**P95/P50时间延迟、并发和稳定性**。




```text
产品设计思路：

1. 产品定义：
   需求分析 -> 先从真实的业务链路里抽取高频场景，完成需求分析和使用场景分析
   产品定义 -> 根据需求，定义模型边界，构建离线评测数据集及评测体系。把样本、标注标准、评测指标、回归流程串成一个闭环。这样的数据集不仅能够评估MVP的效果，还能指导后续版本迭代

2. 产品设计：

3. 产品研发：
   和开发、算法、测试对接，要评估测试

   产品发布上线后的数据分析，推动版本迭代优化
```

# 产品定义

### 需求分析：从真实业务中提取用户的高频场景：
>> 利用 ChatGPT 5.4 爬取公开来源的资料，按意图聚类并输出100条真实案例，划分为20个高频意图标签，并做场景分层。主要目的是为模型边界做准备，明确 “哪些任务用自动化流程可以完成，哪些需要转人工”。

### 产品定义：定义模型边界（也叫任务边界）：
>> 模型边界类似于安全与合规的标准规范，也就是先定义这个系统的职责范围。
>>
>> 边界不是模型会什么，而是产品允许它在什么范围内做决定做回答。

**1.哪些输入问题在Agent的职责范围内，哪些输入问题不在Agent的职责范围内；**

比如是做问答、分类、抽取、总结，还是做多轮 Agent。不同任务，边界完全不一样；比如超出知识范围、缺少必要信息、高风险决策、需要权限验证、需要人工判断的内容。

**2.当输入问题在Agent的职责范围内时，满足哪些约束条件时，Agent才能够输出回答；**

比如必须有足够上下文、必须检索到可信证据、必须拿到正确工具返回结果，否则就不能直接答。

**3.当输入问题不在Agent的职责范围内时，要如何处理这种异常问题，保证合规与安全。**

是拒答、提示信息不足、请求补充材料、转人工，还是降级成保守回答。

### 构建离线评测数据集
>> 上面只是简单爬取了样本，下面进行标注。

*标注体系通常分三层*

```text
第一层：任务标签

• 意图类别
• 是否需要 RAG
• 是否需要 tool
• 是否需要多轮澄清
• 是否应转人工
• 风险等级

第二层：标准答案标签

不是写一个唯一标准答案，而是拆成“回答中必须覆盖的信息点”：
• 关键结论
• 限制条件
• 操作步骤
• 风险提示
• 禁止承诺项
形成一个流程化的规则架构输出回答。

第三层：证据标签

• 正确知识文档 ID
• 正确 chunk / passage
• 若是工具类问题，标正确调用路径或正确返回字段
```

**这样后面如果出了问题，就能把问题拆开评：**

• 是检索没召回

• 还是召回了但生成没用好

• 还是工具调用错了

• 还是本来就不该自动回答

# 评测体系怎么做
我一般把评测分为检索层、决策层、生成层和系统层，四层。

```text
检索层指标：
            Recall@K = Top-K  /   Top-1   /   排序质量

决策层指标：
            工具选择准确率、调用工具后的结果召回率、参数填充准确率、端到端工具成功率

生成层指标：
            答案正确性 -- 是否覆盖标准答案中的关键信息点
            证据一致性 -- 回答是否能被召回证据支撑，有没有脱离知识库自由发挥
            完整性 -- 该说的限制条件、步骤、注意事项有没有说全
            简洁性 / 可执行性 -- 客服场景下不能太啰嗦，回答要让用户能直接执行
            安全性 -- 不能越权承诺，不能给错误退款政策，不能在知识缺失时瞎编
            效率性 -- 生成回答所需时间

系统层指标：
            • p50 / p95 延迟
            • 超时率
            • 并发稳定性
            • 结果波动性
            • 版本回归是否退化
```

• 按照一定规则，写脚本评分：比如关键词覆盖、结构字段完整率
• LLM-as-a-Judge 训练一个裁判LLM
• 人工抽检，专家打分
