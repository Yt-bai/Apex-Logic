## 1. RAG 增强智能客服Agent
我做过一个 RAG 增强智能客服 Agent。它主要想解决的是传统客服场景中“知识命中率低、响应效率低、人工转化率过高”的问题。很多客服问题并不是简单 FAQ，而是需要结合平台规则、上下文和具体业务状态去回答。如果只靠人工，效率低；如果只靠通用模型裸答，又容易出现幻觉，最终既影响用户体验，也浪费客服资源。

痛点：**对话状态的遗忘和内置知识的局限**

在实践上，我先不是急着做模型，而是先定义问题：我从线上高频客服问题中抽取了 100+ 个真实样本，人工标注标准答案要点和对应知识文档，搭了一套离线评测集。然后在方案设计上，我选择 GPT-4o 作为基础模型，设计了 Policy、History、Input、Tool-calling 的 Prompt 结构，再叠加 RAG、多轮记忆管理和工具调用，让它既能检索准确信息，又能处理复杂业务查询。最后我基于评测集持续回归优化，加入 BM25+向量多路召回、去重融合和 Rerank，把召回率从 85% 提升到 90%+，Top-1 证据命中率提升 30%，同时把幻觉率下降了 20%–30%。这段经历让我理解到，Agent 的关键不只是“会回答”，而是能不能围绕真实任务建立可评测、可优化的完整闭环。


```text
- 需求分析
我先看到的问题是，传统pdd客服场景里有两个核心痛点：一是知识命中率低，二是响应效率低。很多问题并不是简单 FAQ，而是要结合规则、上下文和具体业务状态来回答。
如果只靠人工，成本高；如果只靠通用大模型裸答，又容易幻觉，导致人工转化率高、资源浪费。为此，我从线上高频客服问题里抽了 100+ 真实样本，人工标注标准答案要点和对应知识文档，先建立离线评测集，把问题定义清楚。

- 产品定位
我把它定位成一个面向复杂客服场景的知识增强型智能客服 Agent。核心价值是三点：提升回答准确率、替代80%的标准化工作降低转人工率、缩短响应时间。

- 产品设计
方案上，我没有直接做“一个Prompt到底”，而是设计成一套 Agent 链路：
• 用 gpt-4o 做基础模型
• 用 Policy、History、Input、Tool-calling Prompt 管控行为
• 用 RAG 补足知识准确性
• 用记忆管理保证多轮上下文连贯
• 用工具调用处理复杂业务查询
• 用 LangChain + ReAct 组织整个流程
本质上是在做一个“能检索、能理解上下文、还能执行部分查询动作”的客服 Agent。

- 产品落地研发
要明确自己在做的是一个可交付系统。因此，需要重点关注：
第一，不是先堆技术，而是先搭评测集，保证后面优化有明确基准；
第二，把复杂任务拆成模型能力、检索能力、工具调用能力三层，分别落地，再进行编排和记忆。

- 上线后运营/评估
我会重点看四类指标：
• 召回率
• 幻觉率
• 工具调用准确率
• 端到端响应时间
>> 系统已经形成了比较清晰的质量评估框架。工具调用准确率 90%+，召回率 85%，幻觉率下降 20%–30%，平均响应时间 2.5 秒。

- 迭代优化
后续我基于离线评测集持续做回归优化，包括：
• 去重融合
• Rerank 二次排序
最终把召回率提升到90%+，Top-1 证据命中率提升30%。
通过分析评测的数据驱动迭代，而不是凭感觉调参数。
```

## 多步骤文档生成Workflow系统
我做过一个PRD文档生成的 Workflow 系统。它试图解决的是传统大模型在生成专业技术文档时，常见的结构漂移、信息缺失和逻辑不一致问题。很多时候，单轮生成看起来很快，但一旦文档变长、专业性变强，就会出现前后不一致、重点遗漏、格式失控，用户后期修改成本很高。

所以我从产品视角做的不是“换个更强模型”，而是重构生成流程。我把单轮生成拆成了四个环节：结构规划、分段起草、质量校验、汇总输出。在实践中，我选择 Claude 3.5 Sonnet 作为核心模型，设计了 Planning、Drafting、Reviewing、Finalizing 四个节点，用 LangGraph 编排状态机工作流，并加了异常处理，支持错误回退和重试。为了让优化不是凭感觉，我还用 GPT-5.2 制作了 100+ 条技术文档需求数据，人工定义结构模板和必备信息点作为评测标准。最终这个系统的结构完整率提升到 92%，比原来提高了 16%，文档逻辑一致性和信息覆盖度也明显更好。这个项目让我意识到，很多 AI 产品问题不是模型“不够强”，而是产品没有把复杂任务拆成可控流程。

```text
- 需求分析
我发现传统大模型在单轮生成专业技术文档时，很容易出现两个问题：结构漂移和信息缺失。短文本还好，一旦内容长、专业要求高，模型就会出现前后不一致、关键点遗漏、格式失控的问题，后期人工改稿成本很高。

- 产品定位
所以我把这个产品定位成一个面向专业文档场景的流程化生成系统。它不是单纯“帮用户写点文字”，而是要帮用户稳定地产出结构完整、信息覆盖充分、逻辑一致的专业文档。

- 产品设计
核心设计思路是把单轮生成拆成四步：
1. 结构规划
2. 分段起草
3. 质量校验
4. 汇总输出
模型选型上用 Claude 3.5 Sonnet，工作流上设计了 Planning、Drafting、Reviewing、Finalizing 四个节点，并加了 4 个核心 Prompt 约束行为。整体用 LangGraph 编排成状态机，还支持异常处理、回退和最多 3 轮重试。

- 产品落地研发
第一，不是做简单PRD文档生成，而是在做一个可控的生成流程；
第二，自己构建了 100+ 条拥有完整结构的PRD文档作为测评数据，人工定义结构模板和必备信息点，保证这个系统有明确的评估标准。

- 上线后运营/评估
这个系统的核心评估指标不是普通的 BLEU 之类，而是更贴产品结果的指标：
• 平均生成耗时
• 结构完整率
• 逻辑一致性
• 信息覆盖度
结果上，平均生成用时 30–60 秒，结构完整率达到 92%，比原来提升 16%，逻辑一致性和信息覆盖度也显著提升。

- 迭代优化
下一步优化方向也很像产品经理思路：继续加质量校验节点，增强ReAct决策模式，例如
• 结构完整性检查
• 引用真实性检查
• 段落去重
不通过就自动回滚到 Drafting 节点修复。
我的目标不是让模型一次写完，而是先让其自己发现问题并修复。
```

# 自研 Agent 评估与 A/B 测试系统
我做过一个 自研 Agent 评估与 A/B 测试系统。它解决的问题，不是面向最终用户，而是面向 Agent 产品迭代过程中的内部效率问题。因为在做 Agent 时，我发现一个很大的痛点是：评估测试自动化和标准化程度低，每次改模型、改 Prompt、改工作流后，都要花很多时间人工验证，周期长，而且很容易因为评价口径不一致导致上线争议。

所以我把它当成一个“产品基础设施”来做。我设计了一套模块化的 LLM 评估框架，输入是 A/B 两套模型或工作流和测试用例数据集，输出是基于 Accuracy、F1、语义一致性、延迟四项指标的自动化评测报告，形成“提交—自动评测—报告—发布”的闭环。为了兼容不同 Agent 方案，我把推理接口做成标准化 Callable 形式，支持扩展和自定义指标。最终，这套系统把单轮验证周期从 2–3 天压缩到 0.5 天以内，并且在 10+ 并发测试下仍能保证 95% 置信区间的统计准确性。这个经历让我学到，AI 产品经理不只要做前台体验，也要重视后台评测体系，因为没有可持续的评估能力，产品迭代速度和质量都会受限。

```text
- 需求分析
我在做 Agent 项目时发现，真正制约迭代效率的不是“有没有模型”，而是评估测试自动化和标准化程度太低。每次改 Prompt、改模型、改 Workflow，都要人工回归验证，周期长，而且评价口径不统一，导致上线风险和争议都很高。

- 产品定位
所以我把这个项目定位为一个Agent 研发团队内部使用的质量评估平台。它是一个面向研发和产品团队的ToB产品，核心价值是让 Agent 迭代从“拍脑袋上线”变成“数据驱动决策”。

- 产品设计
设计上，我做的是一套模块化 LLM 评估框架：
• 输入是 A/B 两种模型/工作流 + 测试用例集
• 推理接口标准化成 Callable[[str], str]
• 输出是自动化 A/B 测试报告
• 指标包括 Accuracy、F1、延迟
• 整体形成“提交 → 自动评测 → 报告 → 发布”的链路
而且这个框架支持指标扩展和自定义，便于后续版本迭代。

- 产品落地研发
关于Agent如何调用LLM的API，见“调用LLM API.md”文档。

- 上线后运营/评估
指标明确：
• 并发测试用例数量达到 10+
• 统计准确性达到 95% CI
• 单轮验证周期从 2–3 天缩短到 0.5 天以内
这说明它显著提高了迭代效率，降低了回归风险和上线争议。

- 迭代优化
两个方向：
一、接入更多维度指标，比如成本、稳定性、工具调用成功率；
二、针对不同业务场景，沉淀更细的评测模版。

```

# Medical ChatBot 医疗对话机器人版
我做过一个 Medical ChatBot 医疗对话机器人。它想解决的是医疗咨询场景里两个很典型的问题：第一，回答是否足够专业和准确；第二，语气是否足够温和、有同理心。因为医疗咨询和普通问答不一样，用户不仅在意信息对不对，也很在意表达方式是否让人安心。如果模型只追求“答对”，但体验上冷冰冰，用户依然不会信任它。

所以我在实践里把这个项目拆成了“专业性”和“体验感”两个维度来做。模型上我选择 Llama-3.2-3B 作为基座，先通过 SFT 让它适配医疗问答领域，再通过 DPO 去优化回复风格，使它更具同理心。知识层面我加入了基于 LlamaIndex 的 RAG，减少医疗场景下的幻觉风险。为了控制训练成本，我在 SFT 和 DPO 阶段都采用了 LoRA / QLoRA 这种参数高效微调方式。最终，人工评估显示模型在回答专业性和同理心表达上都有提升，不当回答比例下降了 35% 以上。这个项目让我更深地理解到，AI 产品的体验不只是功能正确，还包括情绪价值和信任建立。

```text
- 需求分析
医疗咨询场景里，用户不仅在意“答得对不对”，还在意“说得是否让人安心”。所以我看到的是两个并行需求：
• 提升回答的专业性与准确率
• 优化语气和同理心，改善用户体验
这和普通问答系统很不一样，因为医疗场景对信任要求更高。

- 产品定位
我把它定位成一个面向医疗咨询场景的专业 + 温和型对话机器人。它不是纯知识问答工具，而是一个同时兼顾专业性和情绪体验的 AI 助手。

- 产品设计
整体方案分三层：
• 基座模型：Llama-3.2-3B
• 能力适配：SFT 让模型适应医疗问答
• 风格优化：DPO 提升同理心表达
同时接入 LlamaIndex 构建 RAG，降低幻觉风险，交互层面用 CLI 做原型。
这套设计说明你在产品上明确区分了“知识正确”和“体验更好”两个目标。

-产品落地研发
SFT 阶段用了 ChatDoctor-HealthCareMagic-100k 数据，加系统提示词约束，并用 QLoRA/LoRA 做高效微调；DPO 阶段又用 GPT-5.2 Pro 重写回答，构造偏好对继续做 Preference Learning。
即，不仅做能力增强，也做体验风格调优，而且注意控制训练成本。

- 上线后运营/评估
这类产品的评估不能只看准确率，还要看：
• 回答专业性
• 同理心表达
• 不当回答比例
人工评估显示，专业性和同理心都有提升，不当回答比例下降 35%+。


- 迭代优化
后续可继续往两个方向迭代：
• 对于医疗这种专业性较强的领域，增加风险分级与安全兜底。
• 把 CLI 原型升级成前后端完整的对话产品形态。
```

# 公众号内容策划与后台运营
我在校学生处做过公众号内容策划和后台运营，它本质上解决的是校园信息触达效率和内容互动率的问题。很多校园内容天然重要，但不代表学生愿意看、愿意转发，所以核心问题不是“发出去”，而是“如何让用户愿意消费和传播”。

# 类增量学习研究
我做过一个基于 Adapter 参数微调的类增量学习研究，它试图解决的问题是：模型在持续学习新任务时，容易遗忘旧任务，也就是灾难性遗忘。

我设计了一套orth_loss损失函数，约束不同任务阶段的 adapter 子空间，减少新旧任务之间的相互干扰，并通过指数衰减让这个约束随着任务推进自适应变化。最终这个方法在无范例设定下刷新了SOTA，ImageNet-R 准确率提升1.34%，并在5个公开数据集上验证了泛化能力。

```text
- 需求分析
模型在持续学习新类别时，容易出现灾难性遗忘。

- 研究思路
设计上，我做的是在原有反向传播训练的loss基础上加入新的损失函数：
• 抽取 12 层 Transformer block 中各层 adapter 的上投影线性层
• 与历史 adapter 计算相似度并施加惩罚
• 对正则项加指数衰减系数
目标是实现跨任务子空间解耦，缓解增量遗忘。☆

- 上线后运营/评估
评估上，你有很强的数据支撑：
• 无范例设定下刷新 SOTA
• ImageNet-R 准确率提升 1.34%
• 在 5 个公开数据集上验证泛化能力
这说明方案不只是局部有效，而是有可迁移性。

- 迭代优化
后续可以验证该方法在更多模态、更大模型和更复杂任务序列上的迁移能力。
```
